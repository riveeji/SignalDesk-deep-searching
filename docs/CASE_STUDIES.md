## 案例一：Python Agent 底座选型

- 问题：`如果团队是 Python 技术栈，想做可观测、可接管、适合生产环境的 Agent 系统，LangGraph、PydanticAI 和 Mastra 哪个更适合作为底座？`
- run id：`research_22f8fcc3c7`
- 候选项：`3`
- 来源数：`15`
- 引用数：`35`
- 耗时：`164.5s`
- 结论状态：`grounded`
- 推荐置信度：`high`

这个案例适合用来讲：

- 多候选问题的完整研究闭环
- 覆盖矩阵为什么重要
- 为什么它不是普通搜索，而是带校验的研究流程

## 案例二：单模型适配性评估

- 问题：`DeepSeek 适不适合作为中文技术研究助手的默认推理后端？`
- run id：`research_6ce72c568b`
- 候选项：`1`
- 来源数：`3`
- 引用数：`7`
- 耗时：`133.2s`
- 结论状态：`grounded`
- 推荐置信度：`medium`

这个案例适合用来讲：

- 单目标评估题不需要强行走多候选比较
- Clarifier 如何判断“这是评估题，不是横向排行题”
- 结果如何围绕“是否适合 + 为什么 + 风险”展开

## 案例三：证据不足降级

- 基线 run：`research_0a1068cdbb`
- 操作：人工排除了 `Mastra` 的关键来源，然后重新综合
- 结论状态：`insufficient_evidence`
- 推荐置信度：`low`
- 缺失证据：`Mastra 缺少 官方文档、仓库信号`
- 对题说明：`这是多候选比较题，只有候选项覆盖均衡时才能给确定性推荐。`

这个案例适合用来讲：

- 为什么这个系统不会在证据失衡时继续硬给答案
- `missing_evidence / verdict / confidence` 三个字段分别解决什么问题
- 手动导入来源和来源开关如何参与研究流程

## 当前样本指标

基于当前服务里已成功完成的 `6` 条 run：

- 平均来源数：`9.3`
- 平均引用数：`17.3`
- 平均耗时：`524s`

这个均值包含回归测试和人工插入来源后的重跑任务，所以更适合作为“系统运行样本”而不是“对外宣传指标”。

## 建议你在简历里怎么用

- 对外描述优先使用单个代表性案例，不要直接用全局平均值
- 如果面试官问“有没有量化结果”，优先说：
  - 多候选选型题可稳定生成 `15` 条来源、`35` 条引用
  - 单次研究耗时约 `2~3 分钟`
  - 证据失衡时会自动降级为 `insufficient_evidence`
